Architecture

Enterprise Multi-Tenant OpenClaw on AWS — Zero Modification to OpenClaw

What This Platform Does
Every employee gets their own AI assistant (OpenClaw) running in an isolated Firecracker microVM on AWS. The platform manages who can do what (role-based permissions), what they remember (persistent workspace on S3), and what tools they share (skills with bundled SaaS credentials). All powered by Amazon Bedrock — no API keys, IAM authentication, 10+ models.

The key design principle: OpenClaw runs 100% unmodified inside each microVM. All orchestration — authentication, routing, permission enforcement, workspace sync, audit — happens outside OpenClaw. This means zero fork maintenance, and OpenClaw upgrades are a one-line Docker image rebuild.
Architecture Diagram
🦞
EC2 Gateway (Always-On)
OpenClaw Gateway — Node.js process on port 18789. Maintains long-lived connections to all messaging platforms (WhatsApp, Telegram, Discord, Slack, Feishu). Serves the Web UI. Runs cron scheduler for all tenants.

Tenant Router — Python process on port 8090. Receives messages from Gateway, derives tenant_id (e.g. wa__8613800138000), reads permission profile from SSM, queues requests per-tenant to prevent race conditions, invokes AgentCore Runtime.

Admin Console — Visual management UI for org tree, permissions, approvals, audit logs.
Auth Agent — Handles permission escalation requests with risk assessment and 30-min auto-reject.
⚡
AgentCore Runtime (Serverless)
Each tenant request spins up an isolated Firecracker microVM — not a container, a lightweight VM with its own kernel, filesystem, and network stack. One tenant's prompt injection or memory overflow cannot affect another.

Inside each microVM runs the Agent Container:
entrypoint.sh — Pulls workspace from S3, starts OpenClaw, runs watchdog for periodic sync, handles SIGTERM graceful flush.
server.py — HTTP wrapper that injects Plan A constraints and runs Plan E audit.
OpenClaw — Runs natively, unmodified. Reads SOUL.md, MEMORY.md, loads skills, calls Bedrock.

After response, workspace syncs back to S3. On idle timeout, microVM is released — zero idle cost.
☁️
AWS Services (Right Tool for Each Job)
S3 — Tenant workspaces: SOUL.md, MEMORY.md, daily logs, skills. Incremental sync via aws s3 sync. Shared templates and skills in _shared/ prefix.

SSM Parameter Store — Permission profiles, SOUL templates, Gateway token, Runtime ID. Free, encrypted, hot-reload without redeployment.

Amazon Bedrock — 10+ models (Nova, Claude, DeepSeek, Llama, Kimi). IAM auth, no API keys. Cross-region inference routing (CRIS).

ECR — Agent Container Docker image with pre-installed shared skills.
CloudWatch — Structured JSON audit logs per tenant.
CloudTrail — Every Bedrock API call audited.
DynamoDB (optional) — Multi-turn chat history, approval tokens with TTL.
IAM — Least-privilege role per microVM, S3 path isolation.

S3 Workspace Structure — Per-Tenant File Layout
s3://openclaw-tenants/
├── wa__sarah/ ← Sarah's workspace (Intern, WhatsApp)
│ ├── workspace/
│ │ ├── SOUL.md ← Personality, rules, behavior boundaries (inherited from intern template + personal customization)
│ │ ├── USER.md ← User identity and preferences
│ │ ├── MEMORY.md ← Long-term curated memory (agent writes here)
│ │ ├── memory/
│ │ │ ├── 2026-03-09.md ← Today's auto-generated daily log
│ │ │ ├── 2026-03-08.md ← Yesterday's log (both loaded on startup)
│ │ │ └── ...
│ │ └── skills/ ← Tenant-private skills (if any)
│ └── index/
│ └── memory.sqlite ← Vector search index (rebuilt from Markdown if stale)
├── tg__alex/ ← Alex's workspace (Engineer, Telegram)
│ └── ... (same structure)
└── _shared/ ← Organization-wide shared resources
├── skills/ ← IT-managed shared skills (Jira, S3-files, Slack-bridge...)
├── templates/ ← SOUL.md role templates (intern.md, engineer.md, finance.md...)
└── knowledge/ ← Shared knowledge base (company policies, product docs — read-only)
