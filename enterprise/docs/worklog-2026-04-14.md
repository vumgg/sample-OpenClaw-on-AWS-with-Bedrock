### G5 设计分析：IM 直连 + HEARTBEAT 方案

**核心矛盾：** Telegram Bot API 只允许一个 long-polling client per bot token。4 个 tier 容器不能同时连。

**HEARTBEAT 工作条件：** Gateway scheduler 触发 → agent 执行 → 通过 IM 连接发送结果。Gateway + Agent + IM 发送能力必须协同工作。

**分析了 4 个方案后的结论：**

| 方案 | HEARTBEAT 可行? | 单 bot 体验? | 复杂度 |
|------|----------------|-------------|--------|
| A. 每 tier 一个 dedicated bot | ✓ 完全可行 | ✗ 员工要知道找哪个 bot | 低 |
| B. 独立 IM Gateway 容器转发 | ✗ HEARTBEAT 跨容器不工作 | ✓ | 中 |
| C. EC2 Gateway 继续接 IM | ✗ HEARTBEAT 不工作 | ✓ | 最低 |
| D. webhook + ALB 路由 | ✗ Telegram 只支持一个 webhook URL | ✓ | 高 |

**所有单 bot 方案都有同一个问题：IM 连接在一个地方，agent 在另一个地方，HEARTBEAT 无法跨容器发消息。**

**可行的解决路径：IM Dispatcher + 回调模式**

```
架构：
  IM Dispatcher（持有 bot token，唯一 IM 连接方）
  ├── 接收所有 IM 消息 → 路由到 tier 容器 /invocations
  ├── 暴露 POST /send-message API
  └── 接收 tier 容器的主动推送请求 → 通过 IM 连接发送

  Tier 容器（运行 Gateway + Agent，无 bot token）
  ├── Gateway 运行 HEARTBEAT scheduler
  ├── HEARTBEAT 触发 → agent 执行任务
  ├── 需要发消息给员工 → POST http://dispatcher/send-message
  └── Dispatcher 通过 IM 连接发送
```

**消息流：**
```
入站: Telegram → Dispatcher → tier 容器 /invocations → 回复 → Dispatcher → Telegram
主动推送: tier 容器 HEARTBEAT → agent 执行 → POST Dispatcher /send-message → Telegram
```

**Dispatcher 可以是：**
- EC2 Gateway（最简单，已持有 bot token、已连 Telegram、只需加 /send-message API）
- 独立 Fargate 容器（更干净，但需要新建）

**改动清单（如果用 EC2 Gateway 做 Dispatcher）：**
1. EC2 Gateway 或 Admin Console 新增 `POST /api/v1/internal/send-message` API
2. 该 API 调用 OpenClaw CLI `openclaw send --channel telegram --to {userId} --message {text}`
3. Tier 容器的 workspace_assembler 在 CHANNELS.md 中写入 Dispatcher endpoint
4. OpenClaw Gateway 的 HEARTBEAT 发送逻辑需要读 CHANNELS.md 中的 Dispatcher URL

**待确认：** OpenClaw Gateway 的 HEARTBEAT 发送消息时是直接调 Telegram API，还是通过某个可配置的 hook？如果是直接调 Telegram API，需要看是否有 hook 机制可以重定向到 Dispatcher。这需要进一步研究 OpenClaw 源码。

## Session Storage 移除

**决策：不再使用 AgentCore Session Storage（2026-04-14）。**

解决的 bug：
- 员工身份丢失（缓存了 tenant=unknown 的通用 SOUL）
- KB 文件过期（缓存旧版本，跳过重新下载）
- 1GB 空间溢出风险（output + skills 累积）
- 3-way 状态纠缠（local + Session Storage + S3 不一致）

改为 2-way 状态：microVM 本地 + S3（source of truth）。每次冷启动从 S3 完整组装（+6s）。Fargate 模式下 0s（EFS 持久化）。
